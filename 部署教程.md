# 飞扬排班插件 部署教程（零基础向）

> 目标：把当前 `feiyang-plugin-scaffold` 项目发布到飞书，让俱乐部成员在多维表格里一键调用自动排班。
> 假设读者此前**没有**写过前端、也没用过飞书开发者后台。照着步骤走即可。
>
> 全程只需做四件事：**① 装工具 → ② 开发者后台建应用 → ③ 本地调试跑通 → ④ 上传并发版**。

---

## 步骤零：准备电脑环境

### 0.1 安装 Node.js（≥ 16）

打开浏览器访问 <https://nodejs.org>，下载 **LTS 版**（带"推荐大多数用户"字样的那个），一路下一步安装。

装完后打开**终端**（macOS：启动台搜 `Terminal`；Windows：开始菜单搜 `PowerShell`），
运行下面两条命令验证：

```bash
node -v   # 应输出 v16.x.x 或更高
npm -v    # 应输出 8.x.x 或更高
```

只要不报"command not found"就算成功。

### 0.2 安装飞书开发者 CLI

在终端里敲：

```bash
npm install @lark-opdev/cli@latest -g -f
```

> **macOS 下如果报 `permission denied`**，命令前加 `sudo`：`sudo npm install @lark-opdev/cli@latest -g -f`，会要求输入开机密码。

装完验证：

```bash
opdev --version    # 有版本号输出就 OK
```

### 0.3 登录 CLI

```bash
opdev login
```

终端会让你选环境，**选 `Feishu`**（回车切换、回车确认）。
选完后会自动弹出浏览器登录页，用**管理员飞书账号**扫码或账密登录。
看到终端提示 `login success` 即完成。

> 这一步之后，CLI 就"记得"你是谁了，上传代码无需反复登录。

## 步骤一：在飞书开发者后台创建应用

### 1.1 进入开发者后台

浏览器打开 <https://open.feishu.cn/app>，用**同一个**飞书账号登录。

### 1.2 创建企业自建应用

1. 点击首页 **创建企业自建应用**。
2. 填写：
   - **应用名称**：例如 `飞扬排班助手`
   - **应用描述**：随便写，比如 `大修活动自动化排班`
   - **应用图标**：上传一张 ≥ 240×240 的 PNG，没有的话先用默认
3. 点 **创建**。

### 1.3 记下 App ID

创建成功后，左侧菜单点 **凭证与基础信息**，最上面会有一串 `cli_xxxxxxxxxxxxxxxx`——这就是 **App ID**，**复制下来**。

> 本项目默认的 App ID 是 `cli_a955b62abdf9dcbb`（已写入 `app.json`）。
> 如果你是**在全新的应用里**部署，请用你自己新建应用的 App ID **覆盖 `app.json` 里的那串**。

### 1.4 添加"多维表格插件 / 数据表视图"能力

1. 左侧菜单点 **添加应用能力**。
2. 找到 **多维表格插件**，点进去。
3. 在展开的列表里选 **数据表视图**，点最右边的 **+ 添加**。
4. 进入数据表视图配置页，**最顶上会有一个 `BlockTypeID`**，形如 `blk_xxxxxxxxxxxxxxxxxxxxxxxx`，**复制下来**。

> 本项目默认 BlockTypeID 是 `blk_69df0a371cc10cb4b94145de`（写在 `board_view/block.json`）。
> 同样，**只有在你新建自己的应用时**才需要替换这个值。

### 1.5 申请权限

左侧菜单 **权限管理** → 点 **开通权限**，在搜索框里搜并开通以下几项：

| 权限名 | 说明 |
| --- | --- |
| `bitable:app` | 查看/评论/编辑/管理多维表格（读写都要） |
| `bitable:app:readonly` | 查看多维表格（兜底） |

每项右侧的 **用户身份权限 user_access_token** 打勾。

> 如果企业管理员设置成"免审"，直接生效；否则需要在**步骤四**发版后交管理员审核。

### 1.6 （可选）设置网络白名单

左侧 **安全设置 → 服务器域名白名单**：本插件所有操作都通过飞书 SDK 走，**不对外发请求**，这一栏可以先跳过。

## 步骤二：把本项目关联到你的应用

> 只有当你用的是**自己的**新建应用、而不是俱乐部公用应用时才需要做这一步。

编辑两处：

1. **[`app.json`](./app.json)**
   ```json
   { "appId": "cli_你刚复制的AppID" }
   ```
2. **[`board_view/block.json`](./board_view/block.json)**
   ```json
   {
     "manifestVersion": 1,
     "blockTypeID": "blk_你刚复制的BlockTypeID",
     "projectName": "board_view"
   }
   ```

保存即可，CLI 下次 `upload` 就会投递到新应用。

## 步骤三：本地跑起来（先把功能验收一遍）

> 这一步让你**先在电脑上打开本插件的调试页**，亲眼看到它在多维表格里正常工作后，再发版给别人用。如果公司已经有人在用，想直接跳过调试上线，见步骤四。

### 3.1 安装依赖

```bash
cd "/Users/jerrychen/Code And Project/feiyang-plugin-scaffold/board_view"
npm install
```

> **务必 `cd` 到 `board_view` 目录再运行 npm 命令**，因为 `package.json` 在里面。
> 依赖已存在 `node_modules` 的话可跳过这条。

> ⚠️ **不要运行 `npm audit fix` 或 `npm audit fix --force`！**
> `npm install` 末尾会提示一堆"vulnerabilities"，这些漏洞**全部**来自飞书官方 CLI（`@lark-opdev/cli`）的底层依赖，是**开发期工具**，不会被打包进最终上传的代码产物，对俱乐部使用者完全没有影响。
> 一旦运行 `npm audit fix --force`，它会覆盖/升级 `@lark-opdev/cli`，破坏它的 `postinstall` 恢复机制，导致下次运行 `npm run dev` 报错 `Cannot find module '@bdeefe/feishu-devtools-core/...'`。**直接无视这些提示即可。**

### 3.2 启动调试

```bash
npm run dev
```

几秒后 CLI 会：
1. 调用你的飞书账号，**自动克隆一份演示用多维表格**到你的飞书空间（首次运行）。
2. 把这份表格的 URL 写回 `block.json` 的 `url` 字段。
3. 在默认浏览器里打开该多维表格，并带上 `debugPort=xxxx` 参数。

> ⚠️ **调试期间终端窗口必须一直开着**，`npm run dev` 进程不要 Ctrl+C。
> 飞书的插件 iframe 是实时从 `http://localhost:8080/block/main.js` 加载 JS 的；
> 一旦终端关了/进程停了，插件页面就会变成一片空白（因为资源 404）。
> 这种情况下重新 `npm run dev`，然后回到插件页面按 `Cmd + Shift + R` 硬刷新即可。

### 3.3 在表格里挂载插件

在打开的多维表格页面：

1. 页面左上角"视图"行的末端，点 **+ 新建视图** 或 **添加更多插件**。
2. 滚动到列表底部的 **开发中** 分组，找到形如 `blk_69df0a371cc10cb4b94145de` 的本地插件，点 **+ 添加**。
3. 新 Tab 里就会渲染出 **「大修活动自动化排班」** 界面。

> 如果看不到"开发中"分组：确认 `npm run dev` 还在终端里跑着（出现 `compiled successfully`），然后**刷新整个浏览器 Tab**。

### 3.4 在演示表里做一次端到端验证

**先**在多维表格里建几列字段（如果克隆出来的演示表没有）：

**干事表需要的字段**（字段名要严格一致）：
| 字段名 | 类型 |
| --- | --- |
| 姓名 | 文本 |
| 性别 | 单选（选项：`男`、`女`） |
| 部门 | 多选（至少包含 `维修部`、`研发部` 等） |
| 报名时间段 | 多选（选项就是 `8:00 - 10:00` 这类时段字符串，注意中间是**空格+短横线+空格**） |
| 安排时间段 | 单选（选项与"报名时间段"对齐） |
| 安排位置 | 单选（选项：`1号位`、`4号位`、`5号位`、`6号位`、`机动位`、`学习位`） |

**技术员表**类似，只需要：`姓名` / `报名时间段` / `安排时间段`。

然后在插件里：
1. 顶栏的"权限横幅"应该显示 👑**管理员**（因为你是这份文档的所有者）。
2. 选数据表 → 选排班类型（干事 / 技术员）→ 先勾 **"仅预览，不写回表格"** 点 **执行排班**。
3. 看到底部结果卡片列出各时段人数、点位分配，就说明算法跑通了。
4. 取消勾选"仅预览"，再点一次 **执行排班**，回到表格里刷新，`安排时间段` / `安排位置` 列应已被写入。

至此，本地调试成功。终端按 `Ctrl+C` 关掉 dev server。

## 步骤四：发布到生产（让同事能用）

### 4.1 上传代码包

在 `board_view/` 目录下：

```bash
npm run upload
```

流程：
1. 先 `npm run build`，生成 `board_view/dist/`。
2. 再 `opdev upload ./dist` 将其推到开发者后台。
3. CLI 会提示输入**插件版本号**，用语义化格式，比如第一次发版就填 `0.0.1`；以后每次**必须递增**（`0.0.2`、`0.1.0`、`1.0.0` 都行）。

看到 `Upload success` 就成了。

### 4.2 在开发者后台把"插件"这份版本"装"上去

1. 回到开发者后台 → 你的应用 → 左侧 **添加应用能力 → 数据表视图**。
2. 在页面的 **小组件版本** 下拉里，选刚刚上传的版本号（例如 `0.0.1`）。
3. 上传一个**插件图标**（与应用图标可以不同），填**插件名称**（如"大修排班"）和**一句话介绍**。
4. 点 **保存**。

> 应用图标 = 安装到工作台的图标；插件图标 = 在多维表格"添加视图"列表里显示的图标。两者分开配。

### 4.3 提交应用版本审核

1. 左侧 **应用发布 → 版本管理与发布**。
2. 点 **创建版本**。
3. 依次填：
   - **应用版本号**：比如 `1.0.0`（这是**应用**的版本，与插件版本号独立，不必一致）。
   - **更新说明**：例如"首版，支持干事/技术员自动排班"。
   - **可用范围**：推荐选 **部分成员** → 加入俱乐部干部组/维修部；或直接选"全员"也行。
4. 拉到底点 **保存** → 回到列表点 **申请线上发布**。

提交后交企业管理员审核，一般当天就能过。

> 权限如果在步骤 1.5 时未免审通过，这一步过审后权限才生效。

### 4.4 成员安装使用

审核通过后，有权限的成员打开**任意**一份飞书多维表格：

1. 视图栏点 **+ 新建视图 → 数据表视图**（或 **添加更多插件**）。
2. 在列表的**已发布**分组里找到"大修排班"插件，点 **+ 添加**。
3. 按步骤 3.4 那样选表、选模式、执行即可。

## 日常维护速查

| 场景 | 做什么 |
| --- | --- |
| 改了 `src/` 里的代码 | `cd board_view && npm run dev`（调试）→ `npm run upload`（发版） |
| 算法参数微调（时段/点位容量等） | 改 `board_view/src/config.ts`，版本号 +0.0.1 后 `npm run upload` |
| 想让某个普通成员也能执行排班 | 让他打开插件后把页面上显示的 **用户 ID** 发给管理员；管理员在"授权管理"面板粘贴保存即可 |
| 切换到不同的飞书应用 | 改 `app.json` 的 `appId` + `board_view/block.json` 的 `blockTypeID`，重新 `npm run upload` |
| CLI 登录过期 | `opdev login` 重新登录；或 `opdev whoami` 看当前身份 |

## 常见报错

**`Error: ENOENT: no such file or directory, open '../app.json'`**
→ 你可能是在 `feiyang-plugin-scaffold/` 目录下跑了 npm。`cd board_view` 后重试。

**`Cannot find module '@bdeefe/feishu-devtools-core/libs/config/env'`**
→ `@lark-opdev/cli` 的 `postinstall` 没跑完，或被 `npm audit fix` 破坏。
修复：
1. 在 `board_view/` 下执行 `rm -rf node_modules package-lock.json`
2. 再执行 `npm install`
3. **绝对不要**跟着 npm 提示去跑 `npm audit fix --force`，那会再次打破 CLI 的私有依赖恢复机制。

**`[webpack-cli] Invalid options object. Progress Plugin ...`（说 `name/color/reporters` 不合法）**
→ `webpackbar` 与当前 webpack 5 版本不兼容。本项目已从 `config/webpack.config.js` 里移除 WebpackBar；
如果未来重装后又报这个错，确认一下 WebpackBar 没被加回来即可（它只是控制台进度条，非必须）。

**添加插件后页面一片空白**
→ 99% 是 `npm run dev` 进程已经退出（终端关了 / Ctrl+C 了 / 电脑休眠过），
导致插件 iframe 请求 `http://localhost:8080/block/main.js` 拿到 404。
修复：`cd board_view && npm run dev`，等看到 `Project is running at http://localhost:8080/`，
保持终端开着，然后回到插件页面按 `Cmd + Shift + R` 硬刷新。
排查方法：另开一个终端跑 `lsof -i:8080`，有输出 = 在跑；没输出 = 挂了。

**`TypeError: ESBuildMinifyPlugin is not a constructor`** 或
**`[webpack-cli] Invalid options object. Dev Server has been initialized ...`**
→ 这两个都是 `npm audit fix --force` 把 `esbuild-loader` 升到 4.x、`webpack-dev-server` 升到 5.x 导致的，
新大版本和我们的 webpack 配置不兼容。
修复：打开 `board_view/package.json`，确认两行版本号为：
```
"esbuild-loader": "^2.20.0",
"webpack-dev-server": "^4.10.0",
```
改完后 `rm -rf node_modules package-lock.json && npm install`。再次强调：**别跑 audit fix！**

**`login required` / 上传时 401**
→ `opdev login` 重新登录。如果登了还是报错，`opdev logout` 再 `opdev login`。

**上传成功但在开发者后台看不到新版本**
→ 登录 CLI 用的账号，和创建应用的账号不是**同一人**且不在**同一企业**。切换 `opdev tenant switch` 或重登。

**开发者后台提示"没有这个 BlockTypeID"**
→ `board_view/block.json` 里的 `blockTypeID` 和 `app.json` 里的 `appId` **对不上**：blockTypeID 必须属于那个 appId 对应的应用下、且能力类型必须是"数据表视图"（而不是"记录视图"或"自动化操作"）。

**多维表格里找不到已发布的插件**
→ 应用版本没发布成功，或你的账号不在**可用范围**内。去开发者后台 → 版本管理 → 确认已"发布成功"且你在可用范围里。
